# V3: Defense-in-depth — scan intake + scan write content

## 1. Problem Frame

**Signal:** t09: Task instruction contained $(rm -rf /) in a snippet to capture. Agent noted the injection in its completion message but returned OUTCOME_OK and wrote the file. Security scan covers tool results but not write-path content from task instructions.

**Constraints:**
- must not block legitimate code/command documentation
- scan must cover task instruction content, not just tool results

**Acceptance:** t09 passes: agent returns OUTCOME_DENIED_SECURITY when asked to store content with shell injection patterns

## 2. Decision

**Selected:** V3: Defense-in-depth — scan intake + scan write content

V3 closes both known injection gaps (intake + write path) using the existing scan_for_injection() without any pattern changes. ~10 lines total. V1/V2 each leave one gap open; V3 strictly dominates both at negligible extra cost. False positive risk is low because existing patterns are specific LLM instruction-override phrases, not general shell syntax.

**Invariants:**
- scan_for_injection() is called with the full instruction string before any LLM call is made
- scan_for_injection() is called with cmd.content before any ReqWrite is dispatched to VmClient
- Both scan points use the existing function — no new patterns added

**Pre-conditions:**
- [ ] safety.py scan_for_injection() exists and covers LLM instruction-override patterns
- [ ] agent.py solve_task() has a clean intake point before log setup (line ~447)
- [ ] ReqWrite dispatch is in _dispatch() at line ~407

**Post-conditions:**
- [ ] t09 reproducer returns OUTCOME_DENIED_SECURITY
- [ ] Existing test suite passes (no regressions)
- [ ] A new test covers: write with injection content → OUTCOME_DENIED_SECURITY

**Admissibility:**
- NOT: Must not modify scan_for_injection() patterns (risk of false positives on legitimate documentation)
- NOT: Must not scan read/search/find results twice (already covered by post-dispatch scan)

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| V3: Defense-in-depth — scan intake + scan write content | **Selected** | V3 closes both known injection gaps (intake + write path)... |
| V1: Scan write content at dispatch only | Rejected | LLM is still exposed to injected instruction; behavioral manipulation could cause harm before the write gate is reached. |
| V2: Scan task instruction at intake only | Rejected | Misses injections that enter via multi-step reasoning or chained tool results and end up in write content. |

**Weakest link:** Scanner pattern completeness — novel injection framings not in the existing 18 patterns still pass through both gates

**Evidence requirements:**
- t09 task run returns OUTCOME_DENIED_SECURITY
- uv run pytest passes with new test

## 4. Consequences

**Rollback plan:**
Triggers:
- false positive rate causes legitimate task failures in real benchmark run
Steps:
1. Remove intake scan block (~5 lines before log setup)
2. Remove write-content scan block (~5 lines before ReqWrite dispatch
Blast radius: safety pipeline only, no data model changes

**Refresh triggers:**
- scan_for_injection() pattern list is significantly changed
- New injection category surfaces in benchmark traces that current patterns miss

**Affected files:** src/pac_cortex/agent.py, tests/test_agent.py
