# V6a — Pre-flight prompt assembler with structured JSON output

## 1. Problem Frame

**Signal:** After 4 benchmark rounds the SYSTEM_PROMPT has grown to ~107 lines. ~30 lines (Entity & inbox rules) only apply to t12-t20 but are always loaded — potentially diluting general rule attention on t01-t11 and risking false positives. Separately, AGENTS.md is read for vocabulary but is an agent-planted file with directive content the regex scanner can't semantically classify. Two architectural questions raised: (1) conditional rule loading, (2) LLM-based pre-screening of workspace files before they enter agent context.

**Constraints:**
- No asyncio — system is fully synchronous
- API budget is ~1000 calls/task total (LLM + VM); any extra LLM call costs from this budget
- Must not break existing passing tasks (t01-t11 currently passing, most of t12-t20)

**Acceptance:** Benchmark stays ≥15/20 after change; entity rules correctly fire on t12-t20; no false positives introduced on t01-t11; no increase in OUTCOME_ERR_INTERNAL from budget exhaustion

## 2. Decision

**Selected:** V6a — Pre-flight prompt assembler with structured JSON output

V6a simultaneously achieves high routing quality (informed by actual workspace tree, not just instruction keywords) and high injection resistance (AGENTS.md content can only reach the main agent as sanitized JSON field values — three layers must be pierced simultaneously). It combines the goals of V2 and V4 in a single LLM call with a cleaner security model than either alone. The structured JSON output is the injection firewall: even a successfully injected AGENTS.md entry becomes a string value in {vocabulary: {'term': 'injected text'}}, not an executable instruction. Main agent security rules then prevent acting on vocabulary-framed directives.

**Invariants:**
- _ASSEMBLER_SYSTEM_PROMPT is hardcoded in source — never loaded from workspace or env
- Assembler output is validated as AssembledPrompt (Pydantic) before use — malformed JSON fails closed to _SYSTEM_PROMPT_FULL
- On any pre-flight exception, _SYSTEM_PROMPT_FULL (all sections) is used — never partial or empty prompt
- Pre-flight tree/AGENTS.md reads use vm directly — not subject to the safety pipeline that governs main loop tool results
- api_calls counter includes pre-flight VM + LLM calls — budget guard correctly accounts for them

**Pre-conditions:**
- [ ] _PROMPT_HEADER, _PROMPT_CORE, _PROMPT_ENTITY_INBOX, _PROMPT_SECURITY are defined as module-level constants
- [ ] _SYSTEM_PROMPT_FULL = concatenation of all four sections
- [ ] _ASSEMBLER_SYSTEM_PROMPT is defined as module-level constant
- [ ] AssembledPrompt Pydantic model validated by tests

**Post-conditions:**
- [ ] solve_task() begins with _preflight() call before initialising the log
- [ ] api_calls initialised from preflight_calls return value
- [ ] System prompt passed to main SGR loop is the assembler output (or fallback)
- [ ] 82 tests pass, ruff clean

**Admissibility:**
- NOT: Must not hard-fail if assembler returns unexpected type — fallback required
- NOT: Must not expose AGENTS.md raw content as system instructions
- NOT: Must not skip budget accounting for pre-flight calls

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| V6a — Pre-flight prompt assembler with structured JSON output | **Selected** | V6a simultaneously achieves high routing quality (informe... |
| V6b — free-form prompt output | Rejected | Injected AGENTS.md content propagates verbatim into the main agent system prompt if assembler is fooled — one LLM failure compromises everything. Strictly dominated by V6a on injection resistance. |
| V1 — keyword-based section injection | Rejected | Keyword miss is silent and brittle. No improvement to injection resistance. V6a provides better routing signal (tree structure) at the cost of 1 LLM call, which V6a also pays — but with injection resistance as an additional gain. |
| V5 — status quo | Rejected | Does not address the structural problem. Prompt will continue to grow with each benchmark cycle. Accepted for the competition window only; V6a is the post-competition architecture. |

**Weakest link:** Assembler LLM call reliability — a misclassified task (wrong include_entity_inbox) silently loads wrong rules. Logged at DEBUG level; visible in traces but not surfaced as an error.

**Evidence requirements:**
- Run benchmark against all 20 tasks — score must be >= 15/20
- Verify assembler include_entity_inbox=True fires for t12-t20 tasks in traces
- Verify assembler include_entity_inbox=False fires for t01-t11 tasks in traces

## 4. Consequences

**Rollback plan:**
Triggers:
- Benchmark score drops below 15/20 after V6a deployment
- Assembler misclassification rate > 10% observed in traces
Steps:
1. Revert solve_task() to use _SYSTEM_PROMPT_FULL directly
2. Remove _preflight() call and preflight_calls from api_calls init
3. SYSTEM_PROMPT_FULL already exists as a constant — one-line change
Blast radius: agent.py only — no other modules affected

**Refresh triggers:**
- Benchmark introduces task types not covered by current assembler classification logic
- LLM provider changes structured output behavior
- Score regresses after 2+ consecutive benchmark runs

**Affected files:** src/pac_cortex/agent.py, tests/test_agent.py, tests/conftest.py
