# Expand CONTACT SEARCH → general search_resilience block

## 1. Problem Frame

**Signal:** t19: Agent searched my-invoices/ for "CanalPort Shipping", got no results, immediately returned OUTCOME_NONE_CLARIFICATION. Did not try partial name, customer ID, or listing the directory. Same pattern as t17 (fixed) but for invoices.

**Constraints:**
- search resilience rule from dec-20260327-002 exists but didn't cover this case
- must generalize beyond contacts/ to any resource directory

**Acceptance:** t19 passes: agent finds the invoice by broadening search (list + read fallback)

## 2. Decision

**Selected:** Expand CONTACT SEARCH → general search_resilience block

The CONTACT SEARCH rule was domain-specific; t19 showed the same premature-give-up failure in my-invoices/. A single generalized <search_resilience> block covers all named-entity lookups across any resource directory while preserving the contacts-specific last/first-name hint (validated in t12/t17) as one of four ordered steps.

**Invariants:**
- Ruff: 0 errors
- Pytest: 90/90 pass

**Post-conditions:**
- [ ] Agent tries partial-name search or directory listing before OUTCOME_NONE_CLARIFICATION on any entity type

**Admissibility:**
- NOT: Must not remove the contacts-specific last/first-name step
- NOT: Must not break ruff or pytest

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| Expand CONTACT SEARCH → general search_resilience block | **Selected** | The CONTACT SEARCH rule was domain-specific; t19 showed t... |
| Add INVOICE SEARCH alongside CONTACT SEARCH | Rejected | Rule proliferation — a new gap appears for every new entity type (deals, campaigns, orders). Doesn't scale. |
| Catch-all fallback only, remove per-entity hints | Rejected | Loses the last/first-name split for contacts that was experimentally validated. Regression risk. |

**Evidence requirements:**
- Run t19 and confirm agent attempts at least one partial-name search or directory listing before any failure outcome
