# V1: Local counter in agent.py

## 1. Problem Frame

**Signal:** CLAUDE.md states ~1000 API calls/task budget. Code enforces 30 LLM steps (_MAX_STEPS=30 in agent.py:17) but does not count or cap total API calls (tool calls + LLM calls combined). README claims a "hard guard" that does not exist.

**Constraints:**
- Hard stop must trigger before exceeding the budget, not after
- Budget counter must be visible in run logs for diagnosis
- Must not break the existing _MAX_STEPS guard

**Acceptance:** agent.py tracks total_api_calls (LLM + tool), enforces a configurable hard cap, logs the count per task, and README budget claim matches implementation

## 2. Decision

**Selected:** V1: Local counter in agent.py

Minimum viable enforcement: task-scoped int, zero new abstractions, trivially testable. Current scale (30 steps, ~60 calls/task) makes V2/V3 over-engineering.

**Invariants:**
- api_calls is incremented at every LLM and VM call site in solve_task
- budget check fires before _dispatch, not after
- budget_limit = settings.api_call_budget - _API_BUDGET_MARGIN

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| V1: Local counter in agent.py | **Selected** | Minimum viable enforcement: task-scoped int, zero new abs... |
| V2: call_count on LLMClient + VmClient | Rejected | Requires task-boundary reset logic and changes to two client classes — scope disproportionate to the problem |
| V3: Shared BudgetGuard object | Rejected | Premature abstraction with no second call site; overkill for a single enforcement point |

**Weakest link:** Manual discipline required — future call sites must remember to increment api_calls

## 4. Consequences

**Refresh triggers:**
- _MAX_STEPS raised above 200
- retry counts changed on LLM or VM calls
- parallel task execution introduced

**Affected files:** src/pac_cortex/config.py, src/pac_cortex/agent.py, tests/test_agent.py
