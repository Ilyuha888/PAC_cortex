# Add context tool + entity/inbox sender verification for t12-t20

## 1. Problem Frame

**Signal:** 3 concrete gaps confirmed by code inspection: (1) validate_tool_call() is dead code — written in safety.py:123-130, never imported/called in agent.py; path traversal in tool args goes unchecked. (2) No circuit breaker — injection warnings log and prepend a header but never abort; agent can process unlimited injected files. (3) Security outcome is advisory — agent relies on LLM calling OUTCOME_DENIED_SECURITY; nothing enforces it deterministically.

**Constraints:**
- No new dependencies
- safety.py API unchanged
- ~15 net new lines across 2 files
- All existing tests must pass

**Acceptance:** validate_tool_call imported and called before _dispatch; injection scan aborts task with OUTCOME_DENIED_SECURITY instead of prepending header; 3 new tests pass; all existing tests pass; ruff clean

## 2. Decision

**Selected:** Add context tool + entity/inbox sender verification for t12-t20

The sample agent (pac1-py) confirmed context is a required first step for ERC3 tasks; entity data lives in filesystem files, not RPC; sender verification is the primary scoring differentiator for injection-variant tasks. Implementing all three in one pass (context tool, CRM lookup guidance, identity check rules) minimizes the gap between agent capability and t12-t20 task requirements.

**Invariants:**
- ReqContext passes safety pipeline (validate_tool_call allowlist includes context)
- Sender verification logic lives in SYSTEM_PROMPT, not in agent loop code — prompt governs LLM behavior
- 78 tests remain green after change

**Post-conditions:**
- [ ] vm.context() dispatches to PcmRuntimeClientSync.context(ContextRequest())
- [ ] context is in _ALLOWED_TOOLS frozenset
- [ ] ReqContext appears in NextStep.function discriminated union
- [ ] SYSTEM_PROMPT includes Entity & inbox rules section with 5-step sender verification
- [ ] _SCHEMA_CORRECTION includes context in tool name list
- [ ] 78 tests pass, ruff clean
