# V2 — VmClient._call() with retry + threadpool timeout

## 1. Problem Frame

**Signal:** All 10 VmClient methods are single-shot SDK delegates with no retry logic or timeout. A transient ConnectError wastes one agent step with no client-layer recovery. Competition environment reliability is unknown.

**Constraints:**
- Retry must not exceed the per-task API call budget
- Must not mask permanent errors (auth, not-found) — only retry transient failures
- Must not introduce asyncio (stack is fully sync)

**Acceptance:** VmClient retries transient ConnectError with exponential backoff; permanent errors raise immediately; retry logic is unit-tested with mock SDK

## 2. Decision

**Selected:** V2 — VmClient._call() with retry + threadpool timeout

Correct abstraction boundary: retry and timeout logic lives in VmClient where the SDK calls are made, not in the dispatch layer. Per-call timeout via ThreadPoolExecutor prevents hung gRPC calls from blocking the agent loop indefinitely.

**Invariants:**
- All 10 VmClient methods route through _call()
- Timeout never exceeds vm_call_timeout_s wall-clock seconds per attempt
- ConnectError retried at most vm_call_retries times with exponential backoff
- RuntimeError from timeout is caught in agent._dispatch and formatted as [TOOL ERROR]

**Post-conditions:**
- [ ] 52 tests pass
- [ ] ruff reports no errors
- [ ] vm_call_timeout_s and vm_call_retries are env-overridable via Settings

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| V2 — VmClient._call() with retry + threadpool timeout | **Selected** | Correct abstraction boundary: retry and timeout logic liv... |
| V1 — retry in _dispatch, no timeout | Rejected | No timeout protection; a single hung gRPC call still blocks the process indefinitely. Puts infrastructure logic in the wrong layer (agent.py instead of client.py). |

**Weakest link:** ThreadPoolExecutor timeout cancels future.result() but cannot interrupt a blocking C-level gRPC call — the worker thread may linger until the OS-level socket timeout fires.

**Evidence requirements:**
- smoke test passes with real credentials
- retry fires: VM_CALL_RETRIES=1 VM_CALL_TIMEOUT_S=0.001 logs warnings then raises

## 4. Consequences

**Refresh triggers:**
- gRPC SDK adds native async or cancellation support
- vm_call_timeout_s proves too tight in production (frequent false timeouts)

**Affected files:** src/pac_cortex/config.py, src/pac_cortex/client.py, src/pac_cortex/agent.py
