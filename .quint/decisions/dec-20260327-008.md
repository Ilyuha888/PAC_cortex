# Add attachments to outbox schema + explicit INBOX FILES no-delete rule

## 1. Problem Frame

**Signal:** t19: agent found the correct invoice (INV-004-02.json) but wrote its contents inline in the email body. Grader expected attachments[0]="my-invoices/INV-004-02.json". The outbox schema in _PROMPT_ENTITY_INBOX has no attachments field and no instruction for when to use it.

**Constraints:**
- Must not break existing outbox tasks that don't involve attachments
- Ruff: 0 errors, pytest: 90/90

**Acceptance:** t19 scores 1.0: outbox JSON includes "attachments": ["my-invoices/INV-004-02.json"] when task asks to resend/forward a file

## 2. Decision

**Selected:** Add attachments to outbox schema + explicit INBOX FILES no-delete rule

Two prompt-only fixes in _PROMPT_ENTITY_INBOX: (1) extend outbox schema with optional attachments field and when-to-use guidance; (2) explicit INBOX FILES rule preventing agent from inferring deletion after processing. Both were masked by the earlier attachments failure — uncovered sequentially during t19 verification.

**Post-conditions:**
- [ ] t19 scores 1.00
- [ ] Agent includes attachments field when forwarding files
- [ ] Agent does not delete inbox messages unless explicitly instructed

**Admissibility:**
- NOT: Must not break existing outbox tasks that don't use attachments
- NOT: Must not cause regressions on tasks where delete IS explicitly instructed

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| Add attachments to outbox schema + explicit INBOX FILES no-delete rule | **Selected** | Two prompt-only fixes in _PROMPT_ENTITY_INBOX: (1) extend... |
| Code-level enforcement (validate outbox JSON schema) | Rejected | Prompt fix is sufficient and reversible; code enforcement adds complexity without benefit at this stage. |

## 4. Consequences

**Refresh triggers:**
- New outbox field appears in grader failures
- Inbox delete regression in other tasks

**Affected files:** src/pac_cortex/agent.py
