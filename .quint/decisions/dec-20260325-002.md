# V1: Narrow the catch

## 1. Problem Frame

**Signal:** runner.py:30-34 catches all exceptions with bare `except Exception: logger.exception(...)` and continues to the next task. A crashed task is indistinguishable from a zero-score task in aggregate results, making post-run debugging opaque.

**Constraints:**
- Must not crash the entire session when one task fails
- Must preserve enough context (task_id, exception type, traceback) for post-run diagnosis

**Acceptance:** runner.py records task failures with structured metadata; run summary distinguishes crashed vs completed tasks; existing behavior (continue after failure) is preserved

## 2. Decision

**Selected:** V1: Narrow the catch

Meets the resilience constraint (infra errors continue), maximises failure visibility (logic bugs propagate as crashes), minimal code delta (+4 lines), and gives cmd_run() a clear failure signal via 0-score summary line.

**Invariants:**
- Session must continue past openai.APIError and ConnectError
- Logic bugs (ValueError, AssertionError, RuntimeError) must not be silently swallowed

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| V1: Narrow the catch | **Selected** | Meets the resilience constraint (infra errors continue), ... |
| V2: Re-raise after logging | Rejected | Violates the multi-task resilience constraint — one bad task aborts the entire session |
| V3: Typed TaskResult + threshold abort | Rejected | Complexity delta disproportionate to tactical scope; can be revisited if threshold abort is needed |

**Weakest link:** Exception allowlist completeness — a new infra exception type not in the tuple would crash the session instead of continuing

**Evidence requirements:**
- ruff check passes
- pytest 25/25 green
- ValueError raised inside solve_task() crashes session (manual)

## 4. Consequences

**Refresh triggers:**
- New SDK exception type surfaces in production logs
- Session abort rate rises unexpectedly

**Affected files:** src/pac_cortex/runner.py, src/pac_cortex/main.py
