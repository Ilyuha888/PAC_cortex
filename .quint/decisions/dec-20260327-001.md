# V1 — extra_body passthrough in parse_step()"

## 1. Problem Frame

**Signal:** Gemini 2.5 Flash consumes ~490 "reasoning tokens" internally before producing text output — even for the pre-flight assembler call that returns a tiny JSON (~50 tokens of actual output). This consumed the entire original max_completion_tokens=512 budget, causing pre-flight to fail silently. Workaround: bumped to 4096. Root question: can model-internal reasoning be disabled via API for deterministic/simple structured-output calls?

**Constraints:**
- Must work via OpenAI-compatible API endpoint (used for Gemini and other providers)
- Cannot change LLM provider mid-session
- Pre-flight assembler output is ~50 tokens of JSON — reasoning overhead is disproportionate

**Acceptance:** Pre-flight assembler call succeeds with lower max_completion_tokens OR reasoning tokens are suppressed; actual token usage confirmed in logs

## 2. Decision

**Selected:** V1 — extra_body passthrough in parse_step()"

Minimal blast radius: 2 lines in llm.py, 3 lines in agent.py. No config surface. Provider-agnostic in llm.py — all provider specifics live at the call site. Fully backward-compatible (extra_body=None default).

**Invariants:**
- parse_step() default behavior unchanged (extra_body=None)
- main SGR loop call site in agent.py must not be modified
- llm.py must contain zero provider-specific branching

**Pre-conditions:**
- [ ] Confirm Gemini 2.5 Flash via OpenAI compat accepts extra_body={'thinking_config': {'thinking_budget': 0}}

**Post-conditions:**
- [ ] Pre-flight call passes extra_body={'thinking_config': {'thinking_budget': 0}}
- [ ] usage.completion_tokens_details.reasoning_tokens == 0 on pre-flight (or near-zero)
- [ ] All existing tests pass
- [ ] ruff check src/ tests/ clean

**Admissibility:**
- NOT: Must not change main loop parse_step() call
- NOT: Must not add provider branching to llm.py
- NOT: Must not break existing tests

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| V1 — extra_body passthrough in parse_step()" | **Selected** | Minimal blast radius: 2 lines in llm.py, 3 lines in agent... |
| V2 — thinking_budget setting + parse_step() param | Rejected | Adds config surface and still bakes Gemini key shape into llm.py. Worse portability for more code. |
| V3 — extra_body passthrough + OPENAI_EXTRA_BODY_PREFLIGHT env var | Rejected | JSON-in-env-var is fragile and hard to validate at startup. Over-engineered for a single call-site fix. |

**Weakest link:** extra_body key shape is Gemini-specific — silently a no-op if model swapped to OpenAI o-series without updating the dict

**Evidence requirements:**
- Observe reasoning_tokens in pre-flight usage after change
- Confirm pre-flight still produces valid AssembledPrompt

## 4. Consequences

**Rollback plan:**
Triggers:
- pre-flight starts failing with API errors on extra_body
- model swapped to non-Gemini provider
Steps:
1. Remove extra_body param from parse_step()
2. Remove extra_body kwarg from _preflight() call in agent.py
Blast radius: 2 lines reverted, zero production risk

**Refresh triggers:**
- Model swapped away from Gemini — update extra_body key shape at call site
- OpenAI SDK changes how extra_body is forwarded in beta.chat.completions.parse

**Affected files:** src/pac_cortex/llm.py, src/pac_cortex/agent.py
