# V2: Thin sync adapter (recommended)

## 1. Problem Frame

**Signal:** PAC1-DEV released. Real transport is gRPC/ConnectRPC via `bitgn-api-connectrpc-python`, not REST. Two SDK clients: HarnessServiceClientSync (harness lifecycle) + PcmRuntimeClientSync (per-task filesystem tools). Current BitgnClient (httpx, dict[str,Any]) is entirely wrong transport and has no typed models. Budget is 30 steps/task, not 1000. Python ≥3.14 and openai ≥2.26 required — both mismatched in pyproject.toml. Tool surface: tree, find, search, list, read, write, delete, mkdir, move, answer. Completion via AnswerRequest with typed outcome codes.

**Acceptance:** BitgnClient replaced: uses HarnessServiceClientSync + PcmRuntimeClientSync from bitgn-api-connectrpc-python. All methods return typed Pydantic models. Agent loop calls typed methods with zero raw dicts. pyproject.toml updated to Python ≥3.14 and openai ≥2.26. Budget guard updated to 30 steps. Smoke test passes against PAC1-DEV.

## 2. Decision

**Selected:** V2: Thin sync adapter (recommended)

Satisfies both hard constraints (full API fidelity, full agent isolation via Pydantic-only boundary). Minimal scope (~150 LoC). asyncio.to_thread bridging is acceptable at 30 steps/task — no throughput concern. Pydantic models provide sufficient mock surface for tests without a full Protocol hierarchy.

**Invariants:**
- agent.py and runner.py import zero SDK types — only BitgnClient and its Pydantic models
- Budget hard-stop at 30 steps per task (not 1000)
- All tool results pass through safety module before LLM consumption
- No API key or auth headers — PAC1-DEV is anonymous

**Pre-conditions:**
- [ ] bitgn-api-connectrpc-python installed from buf.build and importable
- [ ] pyproject.toml updated: python >=3.14, openai >=2.26
- [ ] Existing tests pass before rewrite begins

**Post-conditions:**
- [ ] BitgnClient exposes: list_tasks(), start_run(), call_tool(run_id, tool_name, args), submit_answer(run_id, answer_request) — all returning Pydantic models
- [ ] HarnessServiceClientSync handles harness lifecycle (list_tasks, start_run)
- [ ] PcmRuntimeClientSync handles per-task tools (tree/find/search/list/read/write/delete/mkdir/move) and answer submission
- [ ] All 10 tool names are validated against allowlist in call_tool()
- [ ] agent.py budget constant = 30
- [ ] uv run python -m pac_cortex smoke passes against PAC1-DEV

**Admissibility:**
- NOT: SDK types (HarnessServiceClientSync, PcmRuntimeClientSync, any protobuf-generated class) must not appear in agent.py, runner.py, or any test file
- NOT: Raw dict[str, Any] returns from BitgnClient methods are not acceptable
- NOT: httpx must be removed from dependencies after rewrite
- NOT: Budget constant must not remain at 1000

## 3. Rationale

| Variant | Verdict | Reason |
|---------|---------|--------|
| V2: Thin sync adapter (recommended) | **Selected** | Satisfies both hard constraints (full API fidelity, full ... |
| V1: Direct SDK usage (no facade) | Rejected | Fails agent isolation constraint — SDK types leak into agent.py, violating project Pydantic boundary rule |
| V3: Protocol + SDK adapter + fake adapter | Rejected | ~3x scope for one environment. FakeAdapter drift risk outweighs testability gain. V3 is a stepping stone — valid after competition if multi-harness support is needed |

**Weakest link:** Correct SDK method mapping — if any of the 10 tools maps incorrectly to SDK method signatures, agent fails silently at runtime. Must be verified via smoke test against live PAC1-DEV.

**Evidence requirements:**
- Smoke test passes: list_tasks() returns ≥1 task from PAC1-DEV
- call_tool() for each of the 10 tools returns typed ToolResult without error
- No SDK type name appears in agent.py or runner.py (grep check)
