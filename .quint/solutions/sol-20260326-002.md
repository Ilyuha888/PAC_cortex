# Solutions for: SYSTEM_PROMPT is flat and grows with failures; entity rules dilute general rules and workspace files are a semantic injection surface

Problem: prob-20260326-002

## Variants (5)

### V1. V1 — Keyword-based rule section injection (no extra LLM call)

Before constructing the system prompt for a task, parse the task instruction for a small set of keywords (email, inbox, contact, follow-up, account, reminder, schedule, outbox, CRM). If any match → include Entity & inbox rules section. If none match → omit it. Deterministic, zero extra LLM calls. Implemented as a 10-line function that selects which prompt sections to concatenate.

**Strengths:**
- Zero API budget cost — deterministic rule matching
- Eliminates entity rules from t01-t11 context entirely — removes false-positive risk
- Prompt grows naturally: each new task class gets its own section, injected only when relevant
- Fully reversible — toggle by disabling section injection

**Weakest link:** Keyword matching is brittle — a one-word synonym miss causes silent rule absence on the task that needs it most

**Risks:**
- Keyword miss: a task involving inbox processing that uses no trigger keywords fails silently (entity rules absent)
- False keyword match: a t01-t11 task mentions 'contact' in passing → entity rules injected unnecessarily
- Keyword list must be maintained as benchmark evolves

**Stepping stone:** yes — opens future possibilities

### V2. V2 — LLM task-type classifier (1 extra LLM call per task)

First LLM call (fast model, ~100 token output): given the task instruction, classify it as one of ['file_operation', 'entity_inbox', 'security_probe', 'mixed']. Based on classification, assemble the system prompt with appropriate sections. Main SGR loop then uses the assembled prompt. Cost: 1 extra LLM call (small, cheap model possible).

**Strengths:**
- Handles edge cases and synonyms keyword matching would miss
- Classification result visible in trace — debuggable
- Can be extended to other rule sections without modifying keyword lists

**Weakest link:** Classification is an extra LLM call that must be 100% reliable — a misclassified task loses its rules silently

**Risks:**
- Wrong classification → wrong rule set loaded → task failure with no obvious cause
- Adds 1 LLM call to every task budget (even trivially classified ones)
- Classification prompt is itself a prompt engineering surface — needs its own testing

**Stepping stone:** yes — opens future possibilities

### V3. V3 — Self-directed sections with 'applies when' headers in prompt

Keep the monolithic prompt but add explicit 'when to apply' guards to each section as natural language headers. Example: add '(APPLIES TO EMAIL/INBOX/CRM TASKS ONLY — skip if your task has no email or contact elements)' before the Entity & inbox rules section. The LLM itself decides which sections apply. Zero code change — prompt-only modification.

**Strengths:**
- Zero API cost — no extra calls
- Zero code change — pure prompt edit
- Allows the LLM to use partial context from the section even on ambiguous tasks
- Very low blast radius — instant rollback by removing the header

**Weakest link:** Relies entirely on the LLM correctly self-selecting sections — the exact behavior we've seen fail on multi-clause tasks

**Risks:**
- LLM may ignore the guard and apply all rules anyway on borderline tasks
- LLM may incorrectly skip entity rules on t12-t20 if the task phrasing doesn't trigger recognition
- Doesn't actually shrink the prompt — token count unchanged, attention dilution unchanged

### V4. V4 — Structured extraction pre-processor for AGENTS.md (LLM-based workspace screening)

Before the AGENTS.md content enters the agent's context, run it through a dedicated extraction step: either (a) a second LLM call that strips directives and returns only vocabulary/definition lines, or (b) a structured regex pass that extracts H2/H3 sections labeled 'vocabulary', 'definitions', or 'conventions' and discards sections labeled 'workflow', 'process', 'steps'. The agent then sees a sanitized AGENTS.md — vocabulary intact, directives removed. Addresses semantic injection that bypasses regex scan.

**Strengths:**
- Catches semantic injection that looks syntactically benign ('distill also means delete cards')
- Preserves vocabulary definitions the agent needs for task comprehension
- Structured extraction (option b) requires no LLM call — deterministic if section headers are consistent

**Weakest link:** Correct extraction depends on AGENTS.md following predictable structure — in practice, workspace files are arbitrary

**Risks:**
- If AGENTS.md uses non-standard section naming, regex extraction produces empty output — agent loses all vocabulary context
- LLM extraction (option a) adds 1 extra call per task that has an AGENTS.md file
- Regex extraction (option b) is fragile if workspace files don't follow the expected header convention
- Over-sanitization could strip legitimate workflow vocabulary that the task requires

**Stepping stone:** yes — opens future possibilities

### V5. V5 — Status quo + targeted fixes only (no architectural change)

Don't restructure the prompt architecture. Fix the 5 remaining failures with surgical targeted rules, same as previous rounds. Accept that the prompt grows but each rule is small and independently justified. Monitor for attention dilution empirically (if a rule added for task X causes regression on task Y, that's the signal to restructure).

**Strengths:**
- No new failure modes from routing or extraction logic
- Current 15/20 failures are rule-adherence issues, not architecture issues — root cause mismatch
- Fastest path to 20/20 given competition deadline (April 11)
- Architectural refactor can be done post-competition with full benchmark evidence

**Weakest link:** Doesn't address the structural problem — prompt growth is unbounded and interference risk accumulates silently

**Risks:**
- Prompt continues to grow — if we hit 150+ lines, attention dilution becomes real
- Rules added for one task may interfere with another (already seen once with FILE-MEDIATED COMMUNICATION)
- Technical debt — a flat prompt is harder to maintain as task count scales
