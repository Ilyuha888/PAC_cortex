# Solutions for: Safety Hardening — Wire validate_tool_call + fail-fast injection abort

Problem: prob-20260325-004

## Variants (3)

### V1. A: Strict Gate (Fail-Fast)

Wire validate_tool_call before _dispatch; replace advisory injection block with deterministic vm.answer(OUTCOME_DENIED_SECURITY)+return. ~15 net new lines, 2 files.

**Strengths:**
- Deterministic — Python enforces abort, not LLM
- Minimal surface — lowest April 11 risk
- validate_tool_call API already correct, no safety.py changes

**Weakest link:** FP rate of existing injection patterns on competition task content — tested carve-outs exist but untested on unseen tasks

**Risks:**
- FP on legitimate content could abort valid tasks

### V2. B: Warn + Counter (Graduated)

Keep prepend-header behavior but add injection_count; abort only after N warnings (e.g. 3). Softer gate.

**Strengths:**
- Lower FP blast radius — one suspicious line doesn't abort

**Weakest link:** Threshold choice N is arbitrary and attacker-exploitable; grader sees binary outcome

**Risks:**
- N threshold arbitrary; grader scores binary (stopped or didn't); attacker can spread injection across files to stay under threshold

### V3. C: SecurityContext State Machine

Add a SecurityContext dataclass tracking injection_count, blocked_tools, threat_level. Agent checks state before each step. Proper post-competition architecture.

**Strengths:**
- Most expressive — enables graduated response
- Correct long-term architecture

**Weakest link:** Implementation complexity vs deadline — risk of introducing bugs under time pressure

**Risks:**
- Over-engineered for April 11 deadline — 50+ lines, more failure surface

**Stepping stone:** yes — opens future possibilities
