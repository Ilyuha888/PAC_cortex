# Solutions for: VmClient tool calls have no retry or timeout

Problem: prob-20260324-009

## Variants (3)

### V1. Retry in _dispatch (agent.py)

Wrap the existing _dispatch call site in agent.py with a retry loop (2 retries, exponential backoff). Catches ConnectError, sleeps, retries. No timeout — network blips are handled but hangs are not.

**Strengths:**
- Zero changes outside agent.py
- Already the catch site for ConnectError
- Simple loop mirrors LLMClient retry pattern

**Weakest link:** Hung VmClient call still blocks entire solve_task loop indefinitely — retries only help transient failures

**Risks:**
- No timeout — a hung gRPC call blocks indefinitely
- Retry logic mixed into agent control flow

**Rollback:** Revert agent.py changes only

### V2. Retry wrapper in VmClient._call() (client.py)

Add a private _call(fn, *args, **kwargs) method to VmClient in client.py that wraps SDK calls with retry (2 retries, exponential backoff) and a per-call timeout via concurrent.futures.ThreadPoolExecutor.submit().result(timeout=T). All 10 public methods delegate to _call. Add vm_call_timeout_s and vm_call_retries to Settings.

**Strengths:**
- Right abstraction boundary — transport layer owns retry/timeout
- All 10 methods protected with one shared impl
- Timeout via threadpool works in sync + any thread (portable)
- Mirrors _to_dict pattern already in VmClient

**Weakest link:** ThreadPoolExecutor timeout cancels the Python future but cannot kill a blocking gRPC C-extension thread — gRPC call may linger in background

**Risks:**
- ThreadPoolExecutor adds thread overhead per call
- Spawning threads inside a sync SDK call may conflict with SDK internals (ConnectRPC gRPC channel is thread-safe but worth noting)

**Stepping stone:** yes — opens future possibilities

**Rollback:** Revert client.py and config.py changes

### V3. Retry-only wrapper in VmClient, no timeout

Same as V2 but omits the timeout/threadpool complexity. Add _call(fn, *args) to VmClient with ConnectError retry (2 retries, exponential backoff). No timeout mechanism. Add vm_call_retries to Settings.

**Strengths:**
- Clean abstraction, right location
- No threading complexity
- Simpler than V2 — timeout can be added later if SDK exposes it natively

**Weakest link:** No timeout means a single hung call blocks the full task run

**Risks:**
- Still no protection against hung calls
- Tactical half-measure — hung calls remain a risk

**Rollback:** Revert client.py and config.py changes
