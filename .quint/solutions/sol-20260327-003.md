# Solutions for: Agent stores injection payloads in write content instead of refusing

Problem: prob-20260327-004

## Variants (3)

### V1. V1: Scan write content at dispatch only

Before calling _dispatch for ReqWrite commands, run scan_for_injection(cmd.content). If patterns found, call vm.answer(OUTCOME_DENIED_SECURITY) and return early — the write never reaches VmClient.

**Strengths:**
- Directly addresses the failure mode — write is the last gate before data lands on disk
- LLM can still reason about dangerous content in multi-step tasks without early abort
- Zero risk of blocking legitimate tasks at intake for documentation requests

**Weakest link:** LLM exposure window before write gate

**Risks:**
- LLM is still fully exposed to the injected instruction; behavioral manipulation may succeed before reaching write
- Only catches injection if LLM decides to write; a confused LLM might take other harmful actions first

**Rollback:** Remove the pre-dispatch scan block; existing behavior restored in one diff

### V2. V2: Scan task instruction at intake only

At the start of solve_task(), before building the conversation log (line ~447), run scan_for_injection(instruction). If patterns found, immediately call vm.answer(OUTCOME_DENIED_SECURITY) and return — no LLM calls made.

**Strengths:**
- Earliest possible detection; LLM never sees injected instruction
- Zero API budget consumed on malicious tasks
- Simple: 5-line block before log setup

**Weakest link:** Misses mid-session write-path injections that don't appear in the initial instruction

**Risks:**
- Does not cover injections embedded in write content that arrive via multi-step reasoning or chained tool results
- Could generate false positives if a legitimate task description naturally contains LLM override patterns

**Rollback:** Remove the intake scan block

### V3. V3: Defense-in-depth — scan intake + scan write content

Apply both V1 and V2: scan instruction at intake (returns OUTCOME_DENIED_SECURITY before first LLM call), and scan write content before each _dispatch call for ReqWrite (returns OUTCOME_DENIED_SECURITY if content carries injection patterns). Both use existing scan_for_injection() with no pattern changes.

**Strengths:**
- Closes both known gaps simultaneously
- Intake scan prevents LLM exposure to injection; write-content scan acts as final gate
- Uses existing scanner — no new patterns, minimal false positive risk
- ~10 lines total across two insertion points

**Weakest link:** Scanner pattern completeness — patterns cover LLM instruction-override, not arbitrary shell syntax; a novel injection framing not in the pattern list still passes

**Risks:**
- Two scan points means twice the false positive surface, but existing patterns are specific enough to keep risk low

**Stepping stone:** yes — opens future possibilities

**Rollback:** Remove both scan blocks independently; partial rollback to V1 or V2 is clean
