# Solutions for: Agent invariants enforced by prose rules — ordering bugs and prior override

Problem: prob-20260328-001

## Variants (4)

### V1. A — Execution contract: assembler produces task_contract, dispatch enforces it

Extend AssembledPrompt with a `task_contract` sub-model:
```python
class TaskContract(BaseModel):
    inbox_delete_authorized: bool = False
    required_pre_checks: list[str] = []  # e.g. ['filename_scan', 'sender_verify']
    deletion_whitelist: list[str] = []   # explicit paths/patterns
```
The assembler LLM populates this based on task instruction + workspace tree. Dispatch layer in agent.py (or safety.py) validates every ReqDelete against the contract:
- `ReqDelete` on inbox/* when `inbox_delete_authorized=False` → hard block, inject correction message into log.
- `required_pre_checks` validated against a `checks_completed` accumulator before certain actions proceed.

The assembler prompt gains a new output field with clear classification criteria. Prose rules become shorter because code handles enforcement.

Cost: 0 extra API calls (assembler already runs). Assembler prompt grows ~200 tokens for the new output field spec.

**Strengths:**
- Enforcement at code layer — LLM priors cannot override a Python if-statement
- Zero additional API calls
- Assembler already exists as the right abstraction boundary
- Removes need for load-bearing prose sentences like 'Never include a delete step'
- Contract is inspectable in traces for debugging

**Weakest link:** Assembler LLM classification accuracy for task_contract fields. A wrong `inbox_delete_authorized=True` on a non-delete task defeats the mechanism. A wrong `False` on a legitimate delete task blocks correct behavior.

**Risks:**
- Assembler LLM must correctly classify inbox_delete_authorized — wrong classification = wrong enforcement
- Contract fields will grow over time as new invariants discovered — risk of contract bloat
- Over-blocking on false positives (assembler says no-delete but task actually authorizes it)

**Stepping stone:** yes — opens future possibilities

### V2. B — Two-phase execution: mandatory TaskPlan before first tool call

After pre-flight, before any tool call, add a mandatory planning phase. Agent must produce a `TaskPlan` as structured output:
```python
class TaskPlan(BaseModel):
    task_type: Literal['inbox_process', 'distill', 'capture', 'delete', 'email', 'lookup', 'unsupported']
    files_to_delete: list[str]  # must be empty unless task says delete/remove/clear
    files_to_write: list[str]
    security_checks: list[Literal['filename_scan', 'sender_verify', 'account_cross_check']]
    inbox_files_preserved: bool  # code validates: must be True unless task authorizes deletion
```
Plan is validated by code before execution begins. If validation fails, correction injected and agent replans (1 retry). Execution loop then follows the plan.

Cost: 1 extra API call per task.

**Strengths:**
- Forces explicit commitment before any mutation — inbox delete can't 'sneak in' at step 12
- Plan validation is code, not persuasion
- Ordering is enforced: security_checks must be declared upfront
- Visible artifact for debugging — the plan shows what the agent intended

**Weakest link:** Plan-execution drift. The plan is a snapshot at step 0; if tool results reveal unexpected state (e.g. missing files), the agent will deviate. The plan becomes stale but enforcement checked it only once.

**Risks:**
- Plan drift: agent may deviate from plan mid-execution (workspace state changes)
- 1 extra API call per task (user says budget is not sensitive)
- Two structured output schemas (TaskPlan + NextStep) increase model confusion risk
- Agent must correctly classify task_type — same LLM classification risk as variant A

**Stepping stone:** yes — opens future possibilities

### V3. C — Stateful NextStep: safety declarations on every step, code-validated

Extend NextStep with a required `safety_state` sub-model that the agent must fill on every step:
```python
class SafetyState(BaseModel):
    filename_scan_done: bool = False
    sender_verified: bool = False
    inbox_delete_authorized: bool = False
    pending_checks: list[str] = []
```
Dispatch validates safety_state against the action:
- ReqDelete on inbox/* requires `inbox_delete_authorized=True`
- ReqRead on inbox/* requires `filename_scan_done=True` (or the read IS the scan)
- Accumulates `pending_checks` from assembler's required_pre_checks and tracks completion

No extra API calls. Safety declarations are part of every NextStep structured output.

**Strengths:**
- State is visible and validated at EVERY step, not just planning time
- No extra API calls
- Catches ordering violations in real-time (not just at plan time)
- Model must actively maintain state — forces attention to safety

**Weakest link:** LLM self-reporting honesty. The model fills in `filename_scan_done: true` — nothing verifies it actually scanned. A model that wants to skip the scan can just lie. Code validates consistency (delete requires authorized=True) but not truthfulness.

**Risks:**
- Schema noise: model must fill safety fields on every step even for non-inbox tasks
- LLM may hallucinate safety_state values (declare filename_scan_done=True without doing it)
- Increases NextStep token cost by ~50 tokens per step
- Model confusion: dual responsibility (reasoning + safety bookkeeping) in one schema

### V4. D — A+C hybrid: assembler contract + lightweight step-level tracking

Combine A and C with division of labor:
- **Assembler** produces the task_contract (inbox_delete_authorized, required_pre_checks, deletion_whitelist) — the 'what is allowed' question.
- **NextStep** gains a lightweight `checks_completed: list[str]` field — the 'what has been done' question. No full SafetyState, just a string list accumulator.
- **Dispatch** validates both: contract says what's allowed, checks_completed says what's been verified, action is validated against both.

Example flow for inbox processing:
1. Assembler: `{inbox_delete_authorized: false, required_pre_checks: ['filename_scan', 'sender_verify']}`
2. Step 1: agent reads inbox filenames → `checks_completed: ['filename_scan']`
3. Step 2: agent reads email, verifies sender → `checks_completed: ['filename_scan', 'sender_verify']`
4. Step 12: agent tries ReqDelete on inbox → dispatch checks contract → blocked by code.

Cost: 0 extra API calls. NextStep grows by one list field (~20 tokens/step).

**Strengths:**
- Two-layer defense: contract (what's allowed) + tracking (what's done)
- Assembler handles classification (its job), agent handles execution tracking (its job)
- Dispatch-level blocking for the critical invariant (inbox delete)
- Ordering enforcement through checks_completed accumulator
- Minimal schema change to NextStep (one list field, not a full sub-model)
- False positive on contract is caught by traces and fixable in assembler prompt

**Weakest link:** Assembler classification accuracy — same as A. But the blast radius of a wrong contract is lower because checks_completed provides a second signal, and the contract can be overridden by explicit task instruction parsing in code as a fallback.

**Risks:**
- Still depends on assembler LLM for contract classification
- checks_completed is self-reported by the agent (same honesty issue as C, but lower stakes since the hard invariant is contract-enforced)
- Two new concepts for the model to learn (contract + checks_completed)

**Stepping stone:** yes — opens future possibilities
