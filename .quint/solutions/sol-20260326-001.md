# Solutions for: Agent fails 7/20 tasks due to 4 behavioral patterns: premature outcome classification, instruction scope miss, first-plausible-file bias, and inferred cleanup

Problem: prob-20260326-001

## Variants (4)

### V1. V1 — Explore-before-refuse + file-comms distinction

Two targeted rules added to SYSTEM_PROMPT: (1) Before emitting OUTCOME_NONE_UNSUPPORTED or OUTCOME_NONE_CLARIFICATION, always run `tree` and inspect the workspace unless the task explicitly names an external system endpoint (a URL, SMTP server, calendar system). (2) 'Sending email' in a workspace with an outbox/ directory means writing a structured file to outbox/ — that IS a supported file operation. OUTCOME_NONE_UNSUPPORTED applies only when no file-based representation is possible. Addresses t11/t12/t14/t17 (4 tasks).

**Strengths:**
- Directly targets the largest failure cluster (4/7 tasks)
- Rule 1 is general — applies to any future capability-boundary task
- Rule 2 distinguishes file-mediated comms from real network calls — a durable invariant for this workspace class
- Low regression risk: t04/t05/t06/t15 name external endpoints or systems with no file proxy, so they correctly remain NONE_UNSUPPORTED

**Weakest link:** The distinction 'has outbox/ = email is file-mediated' relies on the LLM correctly inferring outbox/ semantics from directory name alone — may fail if the directory is named differently

**Risks:**
- If a future task involves a real SMTP endpoint AND a local outbox, agent might incorrectly use file path
- Rule 1 adds 1-2 extra explore steps on borderline tasks

**Stepping stone:** yes — opens future possibilities

### V2. V2 — Full instruction decomposition checkpoint

Add to SYSTEM_PROMPT operational rules: 'Before taking any action, re-read the task instruction and enumerate EVERY distinct operation it requires (verb + object pairs). Your plan_remaining_steps_brief must cover all of them. Only proceed when your plan accounts for every clause.' This forces the agent to catch secondary requirements like thread linking (t03) and multi-entity updates (t13).

**Strengths:**
- General reasoning improvement — applies to any multi-clause task
- Directly addresses t03 (missed thread link)
- May partially address t13 (forces enumeration of 'reschedule' = possibly multi-file)
- Doesn't change any outcome logic — purely planning discipline

**Weakest link:** LLM may still collapse multi-clause instructions into one step if the secondary clause is semantically similar to the primary (e.g., 'capture and link' → 'capture')

**Risks:**
- May cause over-enumeration: agent invents steps from ambiguous instructions
- Slightly increases context length per step (more in plan field)
- t13 fix is uncertain — depends on LLM correctly inferring that 'reschedule' touches accounts, not just reminders

### V3. V3 — Scope-strict delete rule

Add to SYSTEM_PROMPT: 'Never delete a file unless the task instruction explicitly says to delete or remove it. Do not infer cleanup from AGENTS.md workflows, inbox conventions, or prior steps.' This is a narrowing of the existing STRICT SCOPE RULE specifically for destructive operations, where the cost of a false positive is highest. Addresses t19.

**Strengths:**
- Minimal blast radius — only affects delete decisions
- Perfectly general — applies to any task involving file cleanup
- Very low regression risk: passing tasks with expected deletes (t01, t02) have explicit 'remove/delete' in the instruction
- One sentence, no ambiguity

**Weakest link:** Single-task evidence base (t19 only) — may be coincidence that this grader penalized deletion; other task variants might expect deletion

**Risks:**
- If a task says 'process the inbox' and processing conventionally includes deletion, this rule may cause agent to leave files un-deleted — but t19 shows that's exactly what the grader wanted

### V4. V4 — Combined V1+V2+V3

All three rule additions applied together. Addresses all 7 failing tasks: V1 covers t11/t12/t14/t17; V2 covers t03 and partially t13; V3 covers t19. Net prompt addition: ~8 sentences across 3 existing rule sections.

**Strengths:**
- Maximum coverage across all 4 failure patterns
- Each rule is independently justified — no interaction effects
- Total prompt growth is modest (~8 sentences)
- All rules are general invariants, not task-specific recipes

**Weakest link:** V2's decomposition checkpoint is the weakest — it relies on LLM faithfully enumerating all clauses, which is a reasoning task the model may fail on complex instructions

**Risks:**
- Combined prompt is more complex — increased chance of rule conflicts or LLM ignoring one rule
- V2 may generate verbose plans that inflate context
- More moving parts makes it harder to diagnose if a new failure emerges
