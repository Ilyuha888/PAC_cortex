# Solutions for: Replace httpx stub with gRPC/ConnectRPC client matching real BitGN API

Problem: prob-20260324-006

## Variants (3)

### V1. V1: Direct SDK usage (no facade)

Delete BitgnClient. Runner and agent.py call HarnessServiceClientSync / PcmRuntimeClientSync directly. Tool dispatch calls SDK methods inline. Pydantic models wrap SDK response dicts at call sites.

**Strengths:**
- Minimal new code — no extra layer
- SDK types are the source of truth, no translation

**Weakest link:** Agent isolation — SDK types bleed into agent logic, violating project rule of Pydantic-only boundaries

**Risks:**
- SDK types leak into agent.py and runner.py
- Every test must construct or mock SDK internals
- Changing SDK version touches every call site

**Rollback:** Revert file changes; no dep changes beyond adding SDK

### V2. V2: Thin sync adapter (recommended)

Replace httpx BitgnClient with a new BitgnClient that wraps HarnessServiceClientSync + PcmRuntimeClientSync. All public methods take/return typed Pydantic models (Task, Run, ToolResult, AnswerRequest). SDK types never escape client.py. Agent loop calls asyncio.to_thread() around sync SDK methods. Budget guard updated to 30 steps.

**Strengths:**
- Full API fidelity + full agent isolation
- Minimal blast radius — only client.py is rewritten
- agent.py, runner.py touch only for budget constant and asyncio.to_thread wrapping
- Pydantic models provide natural mock surface for tests

**Weakest link:** Correct SDK method mapping — if tool name → SDK method mapping has gaps, agent silently fails

**Risks:**
- asyncio.to_thread adds thread-pool overhead (acceptable at 30 steps/task)
- Must correctly map all 10 tools to SDK method signatures

**Rollback:** Revert client.py; restore httpx dep; revert budget constant

### V3. V3: Protocol + SDK adapter + fake adapter

Define BitgnClientProtocol (typing.Protocol) with all methods typed. Implement SdkAdapter (wraps real SDK) and FakeAdapter (in-memory filesystem). Agent loop types against the Protocol. Enables full unit tests with no harness.

**Strengths:**
- Full testability — FakeAdapter allows harness-free tests
- Clean dependency inversion — agent loop never imports SDK
- Stepping stone to multi-harness support

**Weakest link:** FakeAdapter drift — fake may diverge from real SDK, creating tests that pass but fail on harness

**Risks:**
- ~3x more code than V2 (protocol file + 2 adapters)
- Over-engineering for a one-environment competition
- FakeAdapter must stay in sync with real SDK semantics

**Stepping stone:** yes — opens future possibilities

**Rollback:** Remove protocol.py + fake_adapter.py; collapse back to V2
