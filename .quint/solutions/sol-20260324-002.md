# Solutions for: No test infrastructure for mocking VmClient and LLMClient

Problem: prob-20260324-007

## Variants (3)

### V1. A: MagicMock(spec=...)

Use unittest.mock.MagicMock(spec=VmClient) and MagicMock(spec=LLMClient) in conftest.py fixtures. Configure return_value on each method to a realistic empty dict. Zero new dependencies — stdlib only.

**Strengths:**
- Interface drift detected: AttributeError if code accesses a removed/renamed method
- Zero new dependencies
- Minimal fixture code (~25 lines)
- Industry standard — pytest + unittest.mock is the canonical pattern

**Weakest link:** Method signature drift (wrong kwargs) is not caught by spec — only attribute name changes are

**Risks:**
- spec only catches attribute access, not method signature changes (wrong kwargs pass silently)
- Return types are still unvalidated — any dict shape is accepted

**Rollback:** Delete conftest.py and test_agent.py

### V2. B: Manual FakeVmClient / FakeLLMClient stubs

Write concrete stub classes that implement the VmClient and LLMClient interfaces explicitly with in-memory dicts. Each method has an explicit signature matching the real class.

**Strengths:**
- Full control over return shapes and call recording
- Type-correct — mypy/ruff can validate stub signatures against real interface
- Tests are explicit about what they return — easier to read

**Weakest link:** Manual sync burden — stubs will drift from real VmClient if methods change and no tooling enforces alignment

**Risks:**
- Stubs must be kept manually in sync with real classes
- More code to maintain (~80-100 lines)
- Stubs can drift silently if not covered by type checking in CI

### V3. C: pytest-mock mocker fixture

Add pytest-mock as a dev dependency. Use the mocker.MagicMock(spec=...) pattern (identical to A functionally, but via pytest-mock's mocker fixture which auto-tears down mocks and provides cleaner API).

**Strengths:**
- Auto-teardown: mocker scope ensures mocks are reset between tests
- Slightly cleaner API: mocker.patch, mocker.spy
- Can spy on real methods without full replacement

**Weakest link:** Marginal functional value over stdlib unittest.mock — the extra dependency is the cost, with minimal return

**Risks:**
- Extra dependency (pytest-mock) for marginal gain over stdlib unittest.mock
- pytest-mock adds nothing that unittest.mock doesn't provide for fixture-scoped mocks
- Team unfamiliarity not relevant here, but dependency surface grows

## Non-Dominated Set

**Pareto front:** A: MagicMock(spec=...), B: Manual FakeVmClient / FakeLLMClient stubs

**Selection policy:** Minimize maintenance burden and dependency overhead. At tactical mode, partial drift detection is acceptable — the real interface is small and stable. B's full detection is not worth 3-4x the code. C adds a dependency with no functional gain over A.

**Recommended:** A: MagicMock(spec=...)
