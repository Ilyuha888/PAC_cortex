# SYSTEM_PROMPT is flat and grows with failures; entity rules dilute general rules and workspace files are a semantic injection surface

## Signal

After 4 benchmark rounds the SYSTEM_PROMPT has grown to ~107 lines. ~30 lines (Entity & inbox rules) only apply to t12-t20 but are always loaded — potentially diluting general rule attention on t01-t11 and risking false positives. Separately, AGENTS.md is read for vocabulary but is an agent-planted file with directive content the regex scanner can't semantically classify. Two architectural questions raised: (1) conditional rule loading, (2) LLM-based pre-screening of workspace files before they enter agent context.

## Constraints

- No asyncio — system is fully synchronous
- API budget is ~1000 calls/task total (LLM + VM); any extra LLM call costs from this budget
- Must not break existing passing tasks (t01-t11 currently passing, most of t12-t20)

## Optimization Targets

- Prompt signal density — only task-relevant rules in context
- Injection resistance — semantic attacks in workspace files caught before agent context
- Implementation simplicity — changes must not increase per-task API budget significantly

## Observation Indicators

- Overall benchmark score — must not regress below 15/20
- False-positive rate on t01-t11 (non-entity tasks)
- Per-task step count — must not increase on average

## Acceptance

Benchmark stays ≥15/20 after change; entity rules correctly fire on t12-t20; no false positives introduced on t01-t11; no increase in OUTCOME_ERR_INTERNAL from budget exhaustion

## Blast Radius

agent.py SYSTEM_PROMPT + optionally safety.py. No DB, no external systems.

## Reversibility

high — prompt changes are fully reversible; safety.py additions can be toggled
