# Agent loop has no structured reasoning checkpoint between tool calls

## Signal

Current agent loop (agent.py:solve_task) runs LLM → tool_calls → execute → repeat with no explicit reasoning step. The LLM's internal reasoning is implicit and unauditable. Stagnation detection (3 repeated calls) is reactive, not preventive. For a one-shot competition scored on correctness and safe tool use, there is no mechanism to catch reasoning drift, wasted calls, or goal confusion before they consume budget. No empirical failures observed — skeleton hasn't run real tasks yet. This is preventive design for a no-second-chance competition.

## Constraints

- ~950 effective API calls per task (1000 minus 50 reserve) — reflection calls eat into this
- Must use openai SDK structured outputs (response_format or tool-based) — no custom constrained decoding
- Must not slow the happy path for simple tasks that need 2-3 tool calls
- Must not break existing tool-use flow — reflection is additive, not replacing tool calls

## Optimization Targets

- Reasoning auditability — every step's rationale is inspectable post-run
- Budget efficiency — fewer wasted tool calls via early course-correction
- Correctness — catch goal drift before it compounds

## Observation Indicators

- Token overhead per reflection step
- Tasks where reflection adds no value (simple 1-2 tool tasks)
- False confidence — reflection says 'on track' but answer is wrong

## Acceptance

Agent emits a Pydantic-typed ReflectionStep before each tool call. Reflection includes: observation from last result, current hypothesis, planned next action with rationale, confidence estimate, budget awareness. Reflection is logged/stored for post-run audit. Reflection can be disabled via config flag for budget-sensitive runs.

## Blast Radius

agent.py (core loop), llm.py (structured output support), config.py (toggle flag), new reflection schema module
