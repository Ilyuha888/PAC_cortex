# VmClient tool calls have no retry or timeout

## Signal

All 10 VmClient methods are single-shot SDK delegates with no retry logic or timeout. A transient ConnectError wastes one agent step with no client-layer recovery. Competition environment reliability is unknown.

## Constraints

- Retry must not exceed the per-task API call budget
- Must not mask permanent errors (auth, not-found) — only retry transient failures
- Must not introduce asyncio (stack is fully sync)

## Optimization Targets

- Transient error recovery without wasting agent steps
- Configurable retry count and timeout via settings

## Acceptance

VmClient retries transient ConnectError with exponential backoff; permanent errors raise immediately; retry logic is unit-tested with mock SDK

## Blast Radius

Any network blip during a task run burns a step and likely causes incorrect tool output. With 30 steps/task budget, even 2-3 wasted steps materially reduce task solve rate.

## Reversibility

medium
