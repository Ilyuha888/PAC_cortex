# Thinking budget vs structured output reliability trade-off

## Signal

Gemini models via litellm cannot produce valid NextStep structured JSON with thinking_budget:0 — the model needs reasoning tokens to organize complex schema output (discriminated union with 12 tool variants). Currently hardcoded at 1024 tokens as a compromise. But: (1) the optimal budget likely varies per step complexity (pre-flight assembler vs mid-task steps), (2) other providers (OpenAI) ignore thinking_config entirely so the setting is provider-specific, (3) we have no telemetry on actual thinking token usage to know if 1024 is right-sized.

## Constraints

- Must not regress benchmark score (22/22)
- Must not increase per-task latency by >20%
- Must work across providers (OpenAI, Gemini, future)

## Optimization Targets

- Minimize wasted thinking tokens per call
- Maximize structured output success rate on first attempt

## Observation Indicators

- Total token cost per benchmark run
- Number of parse retries triggered

## Acceptance

Thinking budget is either (a) adaptive based on provider/step complexity, or (b) validated via telemetry as correctly sized, or (c) removed entirely if providers stabilize structured output without it.

## Blast Radius

config.py, llm.py, agent.py — both parse_step call sites

## Reversibility

high
