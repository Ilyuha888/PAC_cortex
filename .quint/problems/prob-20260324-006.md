# Replace httpx stub with gRPC/ConnectRPC client matching real BitGN API

## Signal

PAC1-DEV released. Real transport is gRPC/ConnectRPC via `bitgn-api-connectrpc-python`, not REST. Two SDK clients: HarnessServiceClientSync (harness lifecycle) + PcmRuntimeClientSync (per-task filesystem tools). Current BitgnClient (httpx, dict[str,Any]) is entirely wrong transport and has no typed models. Budget is 30 steps/task, not 1000. Python ≥3.14 and openai ≥2.26 required — both mismatched in pyproject.toml. Tool surface: tree, find, search, list, read, write, delete, mkdir, move, answer. Completion via AnswerRequest with typed outcome codes.

## Acceptance

BitgnClient replaced: uses HarnessServiceClientSync + PcmRuntimeClientSync from bitgn-api-connectrpc-python. All methods return typed Pydantic models. Agent loop calls typed methods with zero raw dicts. pyproject.toml updated to Python ≥3.14 and openai ≥2.26. Budget guard updated to 30 steps. Smoke test passes against PAC1-DEV.

## Blast Radius

client.py (full rewrite), config.py (remove bitgn_api_key, bitgn_api_url), agent.py (budget constant, tool dispatch), runner.py (task loop), pyproject.toml (deps + python version)

## Reversibility

medium — transport change touches every layer, but PAC1-DEV is the only live environment so there is no production rollback risk
