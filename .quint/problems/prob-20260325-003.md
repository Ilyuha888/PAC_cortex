# Agent not equipped for t12–t20: typed CRM entities, identity verification, and ERC3 runtime variance

## Signal

New tasks t12–t20 use an ERC3-style randomized runtime and introduce typed entity CRM files. The inbox example reveals a class of tasks where the agent must cross-reference sender identity against CRM data before acting — the sender may be spoofed, unauthorized, or legitimate. Current agent has no typed-entity awareness, no identity verification logic, and doesn't fetch current time from the runtime. SDK has minor fixes + time-from-runtime example not yet integrated.

## Constraints

- OUTCOME_DENIED_SECURITY must fire on spoofed or unauthorized senders — not just on prompt injection
- No hard-coding of task patterns — ERC3 variance means rote memorization fails
- Safety pipeline (injection scan, secret redact) must remain intact
- Must not break t1–t11 task handling

## Optimization Targets

- Agent correctly resolves sender identity against CRM data before executing sensitive actions
- Agent integrates runtime time-awareness (SDK example)
- Agent handles typed entity file structures without confusion

## Observation Indicators

- OUTCOME_OK rate on t12–t20 vs t1–t11
- False OUTCOME_DENIED_SECURITY rate (overcautious rejections)

## Acceptance

Agent passes representative t12–t20 tasks in smoke test; identity spoofing case returns OUTCOME_DENIED_SECURITY; legitimate case returns OUTCOME_OK; SDK updated to latest

## Blast Radius

t12–t20 task scoring; identity verification failures could cause security score penalty

## Reversibility

medium — system prompt and dispatch logic changes, no schema migrations
